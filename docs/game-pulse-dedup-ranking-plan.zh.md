# Game Pulse 去重合并与重要性优化计划

更新时间：2026-06-03

## 背景

官方经常会在多个渠道发布同一条信息，例如米游社公告、B站 PV/角色演示、官网公告。后续前端不应把这些当作多条独立资讯刷屏，而应合并成同一个“情报”，只保留多个来源跳转按钮。

同时，外挂封禁名单、违规处罚公示、例行社区治理公告等内容对普通玩家价值较低。建议抓取后默认隐藏或折叠，而不是直接删除，方便后台审计和手动恢复。

## 推荐产品形态

将前端信息流展示单位从单条 `FeedItem` 升级为 `Story`：

- 一个 `Story` 代表同一事件、同一公告或同一视频资讯。
- `Story` 下挂多个 `FeedItem` 来源。
- 前端只显示一张主卡，但保留多个跳转按钮，例如“米游社 / B站 / 官网”。
- 主封面优先级：B站视频封面 > 米游社首图 > 官网 `og:image` > 来源图标 fallback。

## 数据模型建议

长期建议新增：

- `Story`：主标题、摘要、游戏、分类、重要性、展示状态、主封面、发布时间、聚合 hash。
- `StoryItem`：`storyId`、`feedItemId`、来源角色、相似度、匹配方式。

展示状态建议：

- `public`：正常显示。
- `muted`：默认不进主信息流，但可通过筛选查看。
- `hidden`：仅后台可见。

v1.1 可以先不落库 `Story`，先在 API 层临时聚合；但如果要长期上线，建议落库，方便手动合并、拆分和恢复。

## 去重合并策略

采用三层策略，不依赖单一 AI：

1. 规则归一化
   - 去掉标题中的发布时间、括号、特殊符号和渠道后缀。
   - 统一全角/半角、空格、常见游戏名别名。
   - 提取角色名、版本号、活动名、PV 名等核心词。

2. 确定性匹配
   - 同游戏、同时间窗口内，标题归一化后相同或高度相似。
   - 内容共享同一版本号、角色名、活动名。
   - B站标题与米游社标题共享核心名词，例如“角色名 + 演示/PV/前瞻/版本”。

3. AI 辅助匹配
   - 只对规则筛出的少量候选运行。
   - 输出 `sameStory`、`confidence`、`canonicalTitle`、`mergeReason`。
   - 低置信度不自动合并，留给后台人工确认。

时间窗口建议：

- 普通公告：24 小时内候选合并。
- 版本前瞻、PV、角色演示：72 小时内候选合并。
- 热点/爆料类：后续接入热搜后再单独设计。

## 低价值公告过滤

不建议完全不抓取，更建议“抓取后默认隐藏/折叠”：

- 可审计为什么没有展示；
- 可后台恢复；
- 避免规则误伤。

默认 `muted` 或 `hidden` 的典型内容：

- 外挂封禁名单；
- 违规账号处罚公示；
- 例行安全公告；
- 重复客服提醒；
- 与游戏内容无关的社区治理通知；
- 纯工具入口变更且无玩家实际收益的短公告。

初版关键词信号：

- 封禁
- 处罚
- 违规
- 外挂
- 作弊
- 账号安全
- 名单
- 公示
- 社区规范
- 举报处理

关键词只作为候选信号，最终应结合 AI 判断，避免误伤“反作弊系统重大更新”这类真实重要内容。

## 重要性分级建议

当前 `urgent` 容易误导。游戏公告中真正需要“紧急”的内容很少，很多官方写成“重要/紧急”的公告对玩家并不需要立刻处理。

建议前端使用三档：

- `high`
  - 版本更新；
  - 前瞻直播；
  - 新角色/新卡池；
  - 限时活动开启；
  - 大型联动；
  - 长时间服务器异常且影响游玩。

- `medium`
  - 角色 PV / 技能演示；
  - 活动说明；
  - 优化说明；
  - 补偿公告；
  - 官方幕后视频；
  - 实用工具更新。

- `low`
  - 生日贺图；
  - 社区活动；
  - 例行提醒；
  - 小修复；
  - 不影响多数玩家的已知问题。

处理 `urgent` 的建议：

- 数据库枚举可暂时保留，减少迁移风险。
- API 输出时可把 `urgent` 映射为 `high`。
- AI prompt 中不再要求输出 `urgent`。
- 真正紧急情况后续可用 `high + alert=true` 表示，而不是作为普通重要性等级。

## AI Prompt 调整建议

AI 判断标准应从“官方语气是否严肃”改为“普通玩家是否值得立刻知道”。

Prompt 应加入：

- 用户主要关注二游/ACG 资讯；
- 优先识别版本、角色、卡池、PV、活动、福利、维护、补偿；
- 降低封禁名单、社区治理、违规公示的重要性；
- 不要因为标题包含“紧急”“重要”就自动判高；
- 对同源/跨源相似内容输出可用于去重的关键词。

建议输出：

```json
{
  "category": "version | character | event | pv | maintenance | compensation | community | enforcement | other",
  "importance": "low | medium | high",
  "visibility": "public | muted | hidden",
  "summary": "...",
  "reason": "...",
  "dedupKeywords": ["角色名", "版本号", "活动名"]
}
```

## 后端实施顺序

1. 增加 `visibility` 概念，可先放在 `Analysis`，后续再迁到 `Story`。
2. 调整 AI schema 和 prompt，前端不再展示 `urgent`。
3. 增加低价值公告规则标记，再由 AI 复核。
4. 实现 API 层临时 `Story` 聚合。
5. 稳定后新增 `Story` / `StoryItem` 表。
6. 后台增加手动隐藏、恢复、合并、拆分。

## 前端展示建议

情报卡片只展示一张主卡：

- 标题使用 `canonicalTitle`。
- 标签显示游戏、分类、重要性。
- 来源按钮区显示 B站、米游社、官网等多个跳转。
- 展开区显示“该情报已合并 N 个来源”。
- 低价值公告默认不显示，在筛选器中提供“包含低优先级/隐藏公告”。

## 可行性结论

这部分非常适合做成 v1.1。它能解决重复刷屏，让信息流从普通 RSS 列表升级为真正的“情报雷达”，也能为后续接入热搜、NGA、官网、UP 主更新打好结构基础。

建议下个窗口先做 `visibility + importance prompt`，再做 API 层临时聚合，最后再进入前端视觉复现。

## 2026-06-03 临时聚合复查结论：当前规则过宽

用户复查发现：当前 API 层临时聚合会把很多无关信息合并在一起。下一阶段必须先修正这个问题，再继续 UI 细节。

### 新原则

- 宁可暂时不合并，也不要错合并。
- 没有强证据时，不要仅凭标题相似度自动合并。
- 通用词不能作为合并证据，例如：
  - `公告`
  - `资讯`
  - `工具`
  - `更新`
  - `推荐`
  - `生日`
  - `角色`
  - `活动`
  - `官方`
- 跨来源合并必须保留全部来源按钮，不能只保留主来源。

### 建议修正临时聚合规则

短期先保守实现：

1. 必须同游戏。
2. 必须时间接近：
   - 普通公告：12-24 小时；
   - PV/角色演示/版本前瞻：最多 72 小时；
   - 生日图、社区帖、工具更新默认不跨天合并。
3. 必须满足至少一个强条件：
   - 标准化标题完全相同；
   - 外部 ID 或 URL 归一化后相同；
   - AI `dedupKeywords` 中至少 2 个非通用关键词相同；
   - 明确版本号 + 角色名/活动名同时相同。
4. 类别/内容形态必须兼容：
   - `pv` 不要和普通 `announcement` 随意合并；
   - `character` 可与 `pv` 合并，但必须共享具体角色名；
   - `enforcement` 不与任何普通情报合并；
   - `creator_video` 与官方公告合并需更高置信度。
5. 降低或移除 Dice 相似度自动合并阈值；如保留，只能作为候选，不可单独决定合并。

### 必补调试信息

建议临时在 story API 或开发日志中暴露/打印：

- `mergeReason`
- `mergeScore`
- `matchedKeywords`
- `itemCount`
- `sources.length`

这样下个窗口排查“为什么这几条被合并”会快很多。

### 多来源按钮修复要点

- `uniqueStorySources` 应按“来源渠道 + 链接”保留多个入口。
- 同一来源同一 URL 可以去重。
- 不同来源即使标题相同，也应保留独立按钮。
- 前端只负责渲染 `story.sources`，不要在前端重新猜测合并关系。

### 建议测试夹具

下个窗口应先加最小测试或脚本，覆盖：

- 两条不同生日/社区内容不合并。
- 同一角色 PV 的 B站视频和米游社公告可合并。
- 同一版本更新的官网/米游社可合并。
- 外挂封禁名单不进入默认情报流，也不与普通公告合并。
- 合并后的 story 至少保留两个 source jump entries。
