# Game Pulse v1 前端复现与上线计划书

更新时间：2026-06-03

## 目标

先稳定数据流，再复现设计稿，最后上线一个可公开访问的 v1 面板。

v1 的核心不是做一个“看起来完成”的页面，而是做一个真正能长期运行的游戏/ACG 情报雷达：

- 多源采集：米游社官方、B站官方/UP，后续扩展其他来源。
- AI 分析：摘要、分类、重要性判断。
- 实时通知：站内通知优先，邮件后续可选。
- 可视化面板：日/夜模式、毛玻璃模块、封面图、来源图标、游戏筛选。

## 什么时候最适合复现设计稿

建议在“数据流达到可验证状态”后再正式复现设计稿。

最低条件：

- 后端 `/api/public/items` 能稳定返回情报列表。
- 每条情报至少有：
  - `title`
  - `url`
  - `source.type`
  - `source.name`
  - `game`
  - `publishedAt`
  - `createdAt` 或抓取时间
  - `coverUrl`，可为空但字段必须存在
  - `analysis`，可处于 pending/fallback 状态
- `/api/public/sources` 能返回源健康状态。
- `/api/admin/check` 能在 1 分钟内完成一次手动同步。

当前状态：

- 米游社 12 个源已能做到 `checkedSources: 12, failedSources: 0`。
- B站匿名访问仍被风控，B站源暂时禁用。
- 因此可以先做“米游社可用版前端”，B站源以“需要 Cookie / 被风控”状态展示在后台。

最佳复现时机：

- 米游社稳定后即可开始页面骨架、主题、布局、封面展示。
- B站 Cookie 打通后，再补全 B站来源图标、UP 主头像/封面、B站源健康状态。

## 前端复现范围

第一阶段先复现 80% 视觉，不追求像素级一致。

必须做：

- 日间/夜间模式切换。
- 使用 `bg-light.png` 和 `bg-dark.png` 作为页面背景。
- 毛玻璃半透明模块。
- 情报流卡片显示抓取到的封面：
  - 如果 `coverUrl` 有值，显示封面缩略图。
  - 如果没有封面，显示来源类型图标或渐变占位。
- 每条情报显示：
  - 来源图标/来源类型：米游社、B站、官网、RSS、UP 主。
  - 游戏标签。
  - AI 分类标签。
  - 重要性标签。
  - 发布时间：`publishedAt`。
  - 抓取时间：`createdAt` 或后端新增 `fetchedAt`。
- 左侧游戏筛选：
  - 保留真实游戏 Logo/静态图标位。
  - 没有 logo 时显示文字 fallback。
- 右侧摘要：
  - 总情报数。
  - 今日新增。
  - 高重要情报。
  - 源健康。
- 后台入口：
  - 登录。
  - 手动同步。
  - 源列表。
  - B站源 blocked 状态提示。

暂缓：

- 完整图表系统。
- 复杂动画。
- 多用户登录。
- 完整热搜雷达。
- B站动态/投稿之外的复杂交互。

## 封面图策略

后端已经有 `FeedItem.coverUrl`。

前端实现策略：

1. 情报卡片左侧显示封面缩略图。
2. 图片加载失败时 fallback 到来源图标。
3. 米游社封面可能为空，优先从 RSSHub item 的 `enclosure`、`media:content` 或正文首图提取。
4. B站投稿封面应来自视频 `pic` 或 RSSHub 输出。
5. 后续可为 `Source` 增加：
   - `iconUrl`
   - `faviconUrl`
   - `avatarUrl`
   - `gameLogoUrl`

## 第一版上线节奏

### 方案 A：米游社优先上线

适合目标：先把项目真正放到 `pulse.yingzhu.xyz`，可展示、可演示。

预计时间：

- 后端收尾：0.5 天。
- 前端复现设计稿：1-2 天。
- 部署与 HTTPS 反代：0.5-1 天。
- 总计：2-3 天。

上线内容：

- 米游社官方内容。
- 日/夜主题面板。
- 封面/来源/时间/AI 摘要。
- 后台手动同步。
- 源健康状态。
- B站源显示为“待 Cookie / 风控中”。

### 方案 B：B站也打通后上线

适合目标：v1 一开始就覆盖米游社 + B站官方/UP。

预计时间：

- Cookie/RSSHub 验证：0.5-1 天。
- 后端源稳定与后台提示：0.5 天。
- 前端复现设计稿：1-2 天。
- 部署与测试：0.5-1 天。
- 总计：3-5 天。

风险：

- B站 Cookie 会过期。
- B站风控可能随 IP、时间、账号状态变化。
- 上线服务器在新加坡，B站访问稳定性未知。

建议：

先按方案 A 上线，再把 B站作为 v1.1 增强。

## 2026-06-03 用户复查后的前端/产品修正项

当前不应继续只做视觉复现。先修正信息架构和交互问题，再回到视觉细节。

### 情报合并与来源按钮

- 当前临时合并会把部分无关信息合并在一起，属于高优先级正确性问题。
- 前端的“多个来源跳转按钮”依赖后端 `story.sources`，如果后端没正确聚合，同一信息就不会显示多个按钮。
- 下一步应先在 API 响应里验证：
  - `story.items` 是否真的是同一事件；
  - `story.sources` 是否保留了每个来源/渠道；
  - 前端再渲染所有 `story.sources` 的来源图标按钮。

前端验收：

- 一张卡只代表一条真实情报。
- 如果同一情报来自多个渠道，右侧显示多个渠道按钮。
- 如果只有单来源，明确显示“单来源情报”，不要伪装成已合并。

### 顶部导航真实化

当前 `今日摘要 / 情报流 / 源状态 / 后台` 更像视觉占位。下一步需要二选一：

1. 做成真实页面/视图：
   - `今日摘要`：摘要指标 + 精选情报。
   - `情报流`：完整可筛选列表。
   - `源状态`：源健康、最后成功时间、失败原因、Cookie/RSSHub 状态。
   - `后台`：打开或进入管理抽屉/页面。
2. 如果暂时不做多视图，就删除或禁用不工作的 tab，只保留当前可用入口。

不要保留“看起来能点但点了没有语义变化”的导航。

### 右侧登录/后台模块

当前右侧登录栏和后台抽屉功能重复，多个按钮都导向相同内容，用户不知道如何使用。

建议下个窗口先选一个方案：

- 方案 A：公共 v1 直接隐藏右侧登录/后台栏，只保留顶部设置按钮打开后台抽屉。
- 方案 B：右侧栏改为“源状态快捷栏”，只保留：
  - 当前同步状态；
  - 手动同步；
  - 源健康入口；
  - 登录/退出状态。

如果选 B，每个按钮必须有不同且明确的目的，不再全部打开同一个后台抽屉。

### 重要性筛选与计数

当前重要性筛选选中后，其他分类仍显示且条数异常。需要先定义并实现一致规则。

推荐 v1 规则：

- 选中 `高/中/低` 后，情报列表只显示对应重要性。
- 左侧重要性分组可以：
  - 只显示当前选中项和“清除筛选”；或
  - 显示全部分组，但 count 必须是当前其他筛选条件下的真实全量计数，不是当前页局部计数。
- AI 分类和游戏筛选同理，不能混用“当前页计数”和“全局统计”。

技术建议：

- 后端为 `/api/public/stories` 增加 `facets`，返回当前筛选条件下的 `byGame/byCategory/byImportance`。
- 或新增 `/api/public/story-facets`。
- 不建议继续用前端当前页 `stories` 自己算筛选侧栏数量。

### 源状态页最低需求

源状态页应至少展示：

- 来源名称；
- 游戏；
- 类型：米游社/B站/RSS/官网；
- enabled 状态；
- healthStatus；
- lastCheckedAt；
- lastSuccessAt；
- lastError；
- 已抓取条数；
- B站 Cookie/RSSHub 风控提示。

这部分做完后，右侧“源健康”指标才有可解释的入口。
